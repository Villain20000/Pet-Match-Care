# 🌍 Internationalization

Pet-Match & Care ships in two languages: **Greek** (default) and **English**.
Users flip a switch in Settings and the whole UI updates live, AND future
emails go out in the chosen language.

## File layout

```
mobile/src/locales/
  el.ts   ← Greek catalog (source of truth)
  en.ts   ← English catalog (drop-in mirror)
mobile/src/services/i18n.ts
  - useT()  hook
  - useI18nStore  Zustand-backed locale store
  - formatters  date / time / number / relative
  - installLanguageInterceptor  plugs Accept-Language into Axios

backend/src/locales/
  el.ts / en.ts  server catalogs (used by email templates + future i18n
                  for error messages)
backend/src/services/email.service.ts
  - templates.verifyEmail(token, locale)  
  - templates.passwordReset(token, locale)
  - templates.adoptionUpdate({.., locale})  
backend/src/middlewares/auth.ts
  - resolveRequestLocale(req)  reads the JWT.locale claim, falls back to
                                  Accept-Language, default Greek.
```

## Catalog shape

`el.ts` exports a deeply-nested `const el = { login: {…}, dashboard: {…} }`
object. `en.ts` mirrors the same shape with the `Pane` TypeScript helper
(`import type { Pane } from '@/locales/el'`). This means:

- **Missing keys surface as TypeScript errors** at build time.
- New screens should add a sub-tree under their namespace first, then
  add the matching English strings.
- Trying to `t('some.deep.path')` for a missing key falls back to EL,
  then to the key itself as a last-ditch.

## How the runtime works

```ts
import { useT, useI18nStore } from '@/services/i18n';

const t = useT();           // subscribe to locale
t('login.title');          // → 'Είσοδος' or 'Sign in'
t('dashboard.greeting', { name: 'Maria' }); // placeholder substitution

// Switch language (in Settings):
const setLocale = useI18nStore(s => s.setLocale);
await setLocale('en');      // persist to AsyncStorage, PATCH /auth/me
```

Stack header titles update live because every screen reads `useT()` inside
a `useEffect` that re-fires `navigation.setOptions({ title })`.

## Locale detection

First launch:
1. `expo-localization` reads the device locale.
2. If the device starts with `el` → Greek; otherwise English.
3. The first user-controlled change persists to AsyncStorage under
   `@pmc/locale` and overrides device defaults from then on.

The locale is also stored on `User.locale` in the database. When the
user toggles inside the app, the `useI18nStore.setLocale` hook fires a
`PATCH /auth/me` so future server-rendered emails (verification,
password reset, adoption status) follow the device preference.

## Server-side messages

Today the backend answers with English error codes only:

```json
{ "error": "EMAIL_TAKEN" }
```

The mobile app maps every known error code through the catalog before
displaying it:

```ts
api.interceptors.response.use(
  (res) => res,
  (err) => {
    const code = err?.response?.data?.error;
    if (code && i18n.exists(`errors.${code}`)) {
      err.i18nMessage = i18n.t(`errors.${code}`);
    }
    return Promise.reject(err);
  },
);
```

Future work: the `errors` server-side catalog is *also* available via
`/api/_dev/i18n-catalog` for QA to flip between locales in the dev
preview endpoint.

## Email templates

Three built-in templates are locale-aware:

| Template key       | Trigger |
|--------------------|---------|
| `verifyEmail`      | Registration, "resend verification" |
| `passwordReset`    | `POST /api/auth/forgot` |
| `adoptionUpdate`   | Application state transitions |

Each accepts a `locale: 'el' | 'en'` parameter and renders the right
subject + body + web fallback. The user's locale preference is fetched
from `User.locale` on the database so async workers (cron, webhooks)
can route the right language without an `Accept-Language` header.

All user-derived strings go through `escapeHtml()` first; the renderer
loads from the locale catalog only. We treat every input as untrusted.

## Pitfalls / things to watch

- **Municipalities / proper nouns** stay in their original language. The
  `Δήμος Αθηναίων` chip in the dashboard is always Greek regardless of
  locale — these are entity names, not UI strings.
- **Database error fallbacks** — if the server returns an unmapped
  error code, the mobile falls back to `AxiosError.message`.
- **Header titles** re-render only on the screen already mounted. If
  you push one screen, swap language, then *go back*, the prior header
  re-fires its `useEffect` and translates too.
- **Time-based copy** uses `Intl.DateTimeFormat` (`el-GR` vs `en-US`)
  and a lightweight relative-time helper — no `moment` / `date-fns`
  deps.

## CI check (recommended)

Add to a pre-commit hook or CI:

```bash
tsc --noEmit                                # catches missing keys in en.ts
grep -F "t('" mobile/src -r | wc -l         # heuristic: balanced usage
```

A future PR could add a `scripts/check-i18n.mjs` that diffs `Object.keys`
recursively between `el.ts` and `en.ts` and fails if `en` is missing
anything `el` has.
